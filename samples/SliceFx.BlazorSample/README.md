# SliceFx.BlazorSample

Demonstrates the shared-contracts pattern for Blazor WASM: a typed `SliceApiClient` generated by the `slicefx` CLI from the server's route manifest, shared request/response records used directly in both the SliceFx feature handler and the Blazor `EditForm`, and server-side validation errors surfaced as field-level Blazor validation messages.

The sample uses a **standalone** hosting model: the Blazor WASM client and the SliceFx API server run as separate processes connected via CORS.

## What this sample shows

| File | Demonstrates |
|---|---|
| `SliceFx.BlazorSample.Contracts/UserContracts.cs` | Shared contracts project: non-nested request/response records with `[Required]`/`[EmailAddress]` DataAnnotations, mutable `{ get; set; }` properties for `EditForm` two-way binding |
| `SliceFx.BlazorSample.Server/Features/Users/CreateUser.cs` | SliceFx feature using non-nested types from a referenced assembly as its `Handle` parameters |
| `SliceFx.BlazorSample.Server/Features/Users/ListUsers.cs` | GET feature returning `IReadOnlyList<UserSummary>` — included in the typed client as `Users.ListUsersAsync()` |
| `SliceFx.BlazorSample.Server/Program.cs` | CORS policy with `AllowAnyHeader()` — required for preflight on cross-origin requests that carry an `Authorization` header |
| `SliceFx.BlazorSample.Client/SliceApiClient.g.cs` | Pre-committed generated typed client; `(HttpClient)` constructor + `OnRequestPreparing` hook; `SliceApiException.Errors` for Problem Details |
| `SliceFx.BlazorSample.Client/BearerTokenHandler.cs` | `DelegatingHandler` attached via `IHttpClientFactory` to add a bearer token — swap for MSAL or IdentityModel in production |
| `SliceFx.BlazorSample.Client/Program.cs` | `IHttpClientFactory` registration + `new SliceApiClient(factory.CreateClient(...))` wired as a scoped service |
| `SliceFx.BlazorSample.Client/Pages/CreateUser.razor` | `EditForm` + `DataAnnotationsValidator` using Contracts attributes; `SliceApiException.Errors` mapped into `ValidationMessageStore` for server-side field errors |

## Run locally

**Terminal A — API server (port 5101):**

```bash
dotnet run --project samples/SliceFx.BlazorSample/SliceFx.BlazorSample.Server
```

**Terminal B — Blazor WASM client (port 5102):**

```bash
dotnet run --project samples/SliceFx.BlazorSample/SliceFx.BlazorSample.Client
```

Open `http://localhost:5102` in a browser to use the CreateUser form.

You can also call the API directly:

```bash
# Create a user
curl -s -X POST http://localhost:5101/users \
  -H 'Content-Type: application/json' \
  -d '{"name":"Alice","email":"alice@example.com"}'

# List users
curl -s http://localhost:5101/users

# Validation failure — 400 Problem Details with field errors
curl -s -X POST http://localhost:5101/users \
  -H 'Content-Type: application/json' \
  -d '{"name":"A","email":"not-an-email"}'
# → {"errors":{"Name":["The field Name must be a string or array type with a minimum length of '2'."],"Email":["The Email field is not a valid e-mail address."]}}
```

## Regenerate the typed client

`SliceApiClient.g.cs` is committed to source. To regenerate it after changing server routes or Contracts types:

```bash
# 1. Build the server so the route manifest is current
dotnet build samples/SliceFx.BlazorSample/SliceFx.BlazorSample.Server

# 2. Generate the client from the built manifest
dotnet run --project tools/SliceFx.Cli -- client csharp \
  --project samples/SliceFx.BlazorSample/SliceFx.BlazorSample.Server \
  --namespace SliceFx.BlazorSample.Client \
  --output samples/SliceFx.BlazorSample/SliceFx.BlazorSample.Client/SliceApiClient.g.cs \
  --force
```

Or via the MSBuild target (runs the same command, requires the server to be built first):

```bash
dotnet build samples/SliceFx.BlazorSample/SliceFx.BlazorSample.Client -p:RegenSliceClient=true
```

## Notes

**Why mutable `{ get; set; }` on Contracts records?** Blazor's `@bind-Value` writes back to the model property via a setter. Positional record syntax generates `init`-only setters which reject writes at compile time. The explicit `{ get; set; }` form produces a normal setter, while leaving the record semantics (structural equality, `with` expressions) intact.

**Why CORS with `AllowAnyHeader`?** The `BearerTokenHandler` adds an `Authorization` header. Any cross-origin request with a non-simple header triggers an `OPTIONS` preflight. Without `AllowAnyHeader()`, the preflight fails and the browser never sends the actual POST — the failure is visible only in browser devtools, not in `curl` output.

**`dotnet run` only, not `dotnet publish`.** `Microsoft.NET.Sdk.BlazorWebAssembly` enables `PublishTrimmed=true` by default. Trimming would strip the DataAnnotations validation reflection metadata and the JSON reflection used in `ReadFromJsonAsync<T>` and `SliceApiClient.SliceApiException` deserialization. Production trimming configuration is out of scope for this sample.

**Hosted alternative.** If you want a single-process deployment where the server also serves the WASM static files, add `Microsoft.AspNetCore.Components.WebAssembly.Server` to the server project, add a project reference from server to client, and call `app.UseBlazorFrameworkFiles()` + `app.UseStaticFiles()` + `app.MapFallbackToFile("index.html")` in `Program.cs`. The CORS policy and separate client process are not needed in that configuration.
